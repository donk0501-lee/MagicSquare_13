# MagicSquare — Dual-Track RED TDD 작업 보고서 (`feature/dual-track-tdd`)

**문서 목적:** 브랜치 `feature/dual-track-tdd` 전환과, **구현 없이(RED 전용)** Dual-Track 경계·도메인 계약을 실패 테스트로 고정한 작업 내역을 한곳에 보존한다.  
**범위:** 브랜치·산출물 경로·테스트 ID·기대 실패 형태·근거 문서·GREEN 단계에서 채워야 할 모듈 API.  
**비범위:** `magic_square.boundary` / `magic_square.logic` 구현, GREEN·REFACTOR 수행, UI 화면·CI 설정 변경.

**작성 기준일:** 2026-04-28  

---

## 1. 브랜치·저장소 상태

| 항목 | 내용 |
|------|------|

| **의도** | Dual-Track TDD(경계 Track A + 도메인 Track B)를 **RED 먼저**로 고정하기 위한 전용 브랜치 |
| **구현** | `src/magic_square/` 아래 **경계·로직 모듈은 아직 없음** — RED 단계 요구에 따라 의도적으로 미구현 |

## 2. 배경·방법

- 계약 원천은 [`docs/MagicSquare_4x4_TDD.md`](../docs/MagicSquare_4x4_TDD.md) §5(FR)·§6(BR)·§8(Dual-Track)·§9(골든·시나리오)이다.
- 본 작업은 **GREEN·REFACTOR를 수행하지 않았다.** 오직 실패하는 테스트와 테스트 전용 상수만 추가했다.
- Track A는 PRD §8.1의 `code`/`message` 문자열과 정합되도록, 예외 메시지는 `pytest.raises(..., match=...)`로 **바이트 단위 고정 문구**와 매칭되게 작성했다.

## 4. 테스트·검증 결과 (RED 확인)

- 실행: `python -m pytest tests/ -q`
- **기대 결과:** `tests/test_smoke.py`의 패키지 스모크 1건 통과, Track A/B 신규 테스트는 `ModuleNotFoundError: magic_square.boundary` / `magic_square.logic`로 **전부 실패(RED)**.
- 본 보고서 시점에서 이는 “계약 테스트가 구현보다 앞서 있음”을 만족하는 정상 상태이다.

## 5. 테스트 ID와 추적 (요약)

### 5.1 Track A — UI / Boundary

| ID | 보호 계약(요약) |
|----|------------------|
| U-01, U-02 | I1 — 4×4, `SIZE_INVALID` / `Matrix must be 4x4.` |
| U-03, U-04 | I2 — 0 정확히 2개, `BLANK_COUNT_INVALID` |
| U-05, U-06 | I3 — 0 또는 1~16, `VALUE_OUT_OF_RANGE` |
| U-07 | I4 — 비0 유일, `DUPLICATE_NONZERO` |
| U-08, U-09 | 성공 시 `int[6]`, 좌표 1-index (FR-05, BR-06) |

### 5.2 Track B — Logic

| 구간 | 보호 불변조건(요약) |
|------|----------------------|
| 빈칸 좌표 | BR-05, BR-06 — row-major, 1-index (M1/M2 고정 기대) |
| 누락 수 | FR-03, BR-09 — 길이 2, 오름차순 |
| 마방진 판정 | I6, I7, BR-08 — 완성 격자 10선 합 34 |
| `solution` | BR-10~BR-14, §9.1 골든 M1/M2; BR-13·T-05는 `NotMagicError` |
